# Windows U 盘安全弹出辅助工具 — 开发计划

## 目标

在 **Windows** 环境下开发一款程序，解决「无法安全删除硬件 / 此设备正在使用中」等 **原因不明** 的占用问题：把**谁在占、占什么**从黑盒变成可操作的清单，并在用户确认后提供**有限度、可撤销**的缓解手段（关闭句柄、结束进程前提示风险），最终提高 U 盘/移动硬盘能够正常弹出的成功率。

**非目标（第一版不做或仅文档说明）：**

- 保证 100% 弹出（系统服务、防病毒、索引器等场景可能仍需人工处理）。
- 静默强制拔盘（有数据损坏风险，不应作为默认行为）。

## 问题背景

Windows 卷被占用时，`CM_Request_Device_Eject` / shell「弹出」会失败。常见占用来源包括：

- 用户在资源管理器中打开了该卷的文件夹或预览窗格锁定了缩略图。
- 程序打开了卷上的文件或可执行镜像仍驻留内存。
- 工作目录（CWD）或默认保存路径指向 U 盘上的某进程。
- 杀毒、备份、云同步、索引（Windows Search）后台扫描。
- 卷级或路径级的独占打开（少见但会导致整卷无法卸载）。

本工具的核心价值是：**枚举与卷相关的句柄与进程**，交叉引用得到「进程 → 路径/句柄类型」，让用户能针对性关闭窗口或结束进程，而不是盲目重启。

## 技术路线与语言选择

阶段 1 已经包含系统托盘常驻、开机自启动、设备插入/移除监听、一键弹出和占用扫描，因此语言选择不能只看「调用 API 快不快」，还要看 Windows 桌面集成、分发、权限模型和长期维护。

### 推荐主语言：C# / .NET 8+

C# 更适合作为本项目的主语言，尤其适合第一版就要做成 Windows 常驻工具的目标。

- **Windows API 集成自然**：P/Invoke、WMI、注册表、Windows 消息、设备通知、Restart Manager、SetupAPI / cfgmgr32 都有成熟资料和示例。
- **托盘与 GUI 更稳**：可选择 WinForms 托盘 MVP、WPF 桌面界面，后续也可迁移到 WinUI 3；不必为 Python GUI 打包和托盘兼容性消耗太多精力。
- **自启动与设置管理直接**：操作 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`、应用配置、通知区域图标都比较顺手。
- **打包分发更符合 Windows 用户预期**：可发布单文件 exe、MSIX 或安装包；代码签名、版本信息、图标、卸载逻辑更容易标准化。
- **高权限边界更清晰**：普通进程、管理员重启、UAC 提权、受限操作失败原因更容易做成稳定的用户体验。

结论：**以 C# / .NET 8 作为主实现语言**。第一版可用 WinForms 做轻量托盘和窗口，降低 UI 成本；若后续需要更现代的界面，再引入 WPF 或 WinUI 3。

### Python 的合理定位：验证和工具脚本

Python 仍然适合做早期 API 验证、句柄枚举实验和测试脚本，但不建议作为最终常驻桌面程序的主语言。

- 优点：开发快，`ctypes` / PyWin32 / WMI 调用方便，适合快速验证「能否定位占用进程」。
- 缺点：托盘常驻、GUI、打包、杀毒误报、管理员重启、代码签名和用户安装体验更难控制。
- 建议用途：保留 `tools/` 下的验证脚本，用于探索 WinAPI 行为；不要把 Python 原型直接等同于正式产品。

### C++ 的合理定位：底层能力补充

C++ 适合处理最底层、最敏感的 Windows 能力，但不适合承担整个产品。

- 适合场景：如果 .NET P/Invoke 在系统句柄枚举、路径反查、超时控制或性能上遇到瓶颈，可以封装一个很小的 native helper。
- 不适合场景：托盘 UI、设置、自启动、日志、交互流程不需要 C++ 承担。
- 原则：只有在 .NET 用户态能力不足时才引入，避免过早增加构建、签名和崩溃排查成本。

### Rust / Go 的定位

Rust 和 Go 都可以做稳定 CLI 或后台工具，但对本项目不是最优主语言。

- **Rust**：安全性和单文件分发优秀，但 Windows 桌面、COM/WMI、SetupAPI、托盘生态的开发成本更高。
- **Go**：分发简单，适合 CLI/后台服务，但 Windows 原生桌面体验、复杂 WinAPI 绑定和托盘 UI 不如 .NET 顺手。

### 语言选择结论

```text
C# / .NET 8+: 主程序、托盘、GUI、自启动、设备识别、安全弹出、权限流程
Python: 早期验证脚本、实验性句柄枚举、测试辅助
C++: 必要时作为小型 native helper，补齐 .NET 难以稳定处理的底层能力
SQLite 或 JSON 配置: 本地设置、日志和设备历史记录
```

## 功能规划

### 阶段 1：后台识别与只读诊断（MVP）

第一阶段就需要接近 Windows 自带「安全删除硬件并弹出媒体」的使用模式：程序可常驻后台，自动识别插入的 U 盘/移动硬盘，并允许用户对某个可移动盘执行「弹出」或「扫描占用」。

1. 提供设置项：是否开机自启动。
   - 默认不自动开启，由用户在设置中明确打开。
   - 第一版优先使用当前用户级启动项（如注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 或启动文件夹），避免为了自启动要求管理员权限。
2. 自动识别可移动存储设备：
   - 监听设备插入/移除事件，更新当前可弹出的盘符列表。
   - 展示盘符、卷标、容量、文件系统、设备名称等基础信息。
3. 对每个可移动盘提供操作：
   - 「弹出」：调用 Windows 安全弹出相关 API，成功后提示可拔出。
   - 「扫描占用」：当弹出失败或用户主动请求时，扫描哪个进程正在占用该硬盘。
   - 「刷新」：重新枚举设备和占用状态。
4. 解析盘符到卷设备路径（如 `\\.\E:`、`\\?\Volume{guid}\`）。
5. 列出：
   - 该卷上的打开句柄关联进程（PID、进程名、命令行若有）。
   - 尽量解析到用户可见路径（规范化到 `E:\...`）。
6. 以系统托盘菜单或简单窗口展示设备列表和占用表格；支持导出 JSON/文本便于反馈问题。

**验收：** 插入 U 盘后程序能自动出现在设备列表；用户可对该盘点击弹出；若弹出失败，在「资源管理器打开 U 盘某文件」与「某记事本打开 txt」等可控场景下，能稳定指向正确进程。

### 阶段 2：辅助释放（需谨慎）

在用户**明确确认**后提供：

- 「刷新」重新扫描。
- 「在资源管理器中打开进程所在目录」（方便用户手动关文件）。
- 可选：**结束选定进程**（强提示数据丢失风险、排除系统关键进程列表）。
- 可选：调用 **Restart Manager** 相关 API，让部分应用优雅释放文件（适用面有限，可作为增强）。

**原则：** 默认不杀进程；杀进程前二次确认；记录操作日志。

### 阶段 3：体验与覆盖

- 优化系统托盘体验：设备分组、状态图标、弹出成功/失败通知。
- 常见误报处理：系统进程、`System` PID、暂不可解析句柄的说明文案。
- 安装包：单目录绿色版优先（`pip`/zip 或 PyInstaller 视分发策略而定）。

## 关键技术要点（实现备忘）

| 主题 | 方向 |
|------|------|
| 枚举句柄 | 由内核/用户态完整枚举句柄成本较高；可结合 `NtQuerySystemInformation(SystemHandleInformation)` 思路或受信的现成实现；注意权限与 32/64 位一致性。 |
| 权限 | 部分信息需 **管理员** 运行；程序启动时检测并提示「提升权限可看到更多占用」。 |
| 路径解析 | `GetFinalPathNameByHandleW`、Dos 设备名映射；注意 `\Device\HarddiskVolumeN` 与盘符对应。 |
| 设备识别 | 使用 WMI 事件、`WM_DEVICECHANGE` 或轮询作为降级方案；区分真正的可移动盘、USB 移动硬盘、读卡器和固定硬盘。 |
| 开机自启动 | 优先使用用户级启动项；设置中必须能开启/关闭；卸载或退出常驻时不能留下无效启动项。 |
| 与安全弹出关系 | 诊断清楚后，用户关闭占用源再执行系统「弹出」；程序可封装**重试弹出**（调用 `SetupDi` / cfgmgr32 或 shell API）作为最后一步。 |

## 提权与高权限操作注意事项

本工具会触碰 Windows 的进程、句柄、卷和设备管理能力，必须把「哪些操作需要提权、为什么需要、失败时如何降级」设计清楚。原则是：**普通权限优先，只在用户明确选择高权限功能时请求管理员权限**。

### 权限分层

1. **普通用户权限**
   - 枚举当前用户可见的进程与部分句柄信息。
   - 查询盘符、卷 GUID、基本文件系统信息。
   - 展示诊断结果、导出日志、刷新扫描。
   - 尝试安全弹出可能会失败，但应给出明确原因和下一步建议。

2. **管理员权限**
   - 查看其他用户会话或更高完整性级别进程的详细信息。
   - 对部分进程执行句柄复制、路径反查或关闭动作。
   - 结束非当前用户进程。
   - 调用设备管理相关 API 执行卸载、弹出、禁用/启用设备等操作。

3. **调试权限（`SeDebugPrivilege`）**
   - 可能用于打开系统级或受保护进程、复制其句柄。
   - 风险较高，容易触发杀毒/EDR 告警。
   - 应作为显式高级模式启用，而不是程序启动时默认启用。

### 值得特别注意的提权点

| 操作 | 可能需要的权限 | 风险 | 建议 |
|------|----------------|------|------|
| 枚举全系统句柄 | 管理员 / `SeDebugPrivilege` | 被安全软件识别为敏感行为；部分句柄无法解析或阻塞 | 默认先普通扫描；高级扫描前说明用途和范围。 |
| 复制其他进程句柄并解析路径 | 管理员 / `SeDebugPrivilege` | 访问失败、句柄类型不兼容、目标进程退出导致竞态 | 设置超时与错误分类；失败时显示「不可解析」而不是崩溃。 |
| 关闭其他进程持有的句柄 | 管理员 / `SeDebugPrivilege` | 极易导致目标程序异常、数据损坏或崩溃 | 第一版不建议实现；如实现，必须隐藏在高级选项并二次确认。 |
| 结束进程 | 当前用户或管理员 | 未保存数据丢失；结束系统/安全软件进程可能影响系统稳定 | 默认只建议用户手动关闭；程序结束进程前显示 PID、路径、命令行和风险提示。 |
| 调用 Restart Manager 关闭应用 | 普通用户或管理员，取决于目标进程 | 应用不一定支持优雅关闭；可能触发保存提示 | 优先于强杀进程；记录返回码和失败原因。 |
| 重试安全弹出 / 卸载卷 | 普通用户可能可用，管理员覆盖面更高 | 卷仍有写缓存或后台 IO 时可能失败；强制卸载有数据风险 | 仅在诊断为空或用户确认后执行；不要默认强制卸载。 |
| 禁用/启用 USB 设备 | 管理员 | 可能影响同一控制器下其他设备；误操作会断开键鼠/网卡等 | 不作为 MVP 功能；如后续支持，必须精确定位设备实例 ID。 |
| 安装服务或驱动 | 管理员 / 驱动签名 | 攻击面、签名、卸载、兼容性和安全审查成本高 | 第一版避免服务化和内核驱动；只有用户态能力不足时再评估。 |

### UX 与安全约束

- **不要启动即要求管理员权限**：先用普通权限给出可用诊断，再提示「以管理员重新扫描可看到更多占用」。
- **所有高风险操作必须可解释**：界面显示将操作的 PID、进程名、可执行路径、命令行、占用文件路径和预计影响。
- **二次确认**：结束进程、关闭句柄、弹出/卸载设备等操作需要二次确认；默认按钮应是取消。
- **关键进程保护**：对 `System`、`csrss.exe`、`wininit.exe`、`services.exe`、杀毒/EDR、存储驱动相关进程等默认禁止直接操作，只提供说明。
- **日志脱敏**：导出日志可能包含用户文件名、路径、用户名；需要提示隐私风险，并提供脱敏选项。
- **权限失败要降级**：权限不足时不要报错退出，应展示「当前权限可见范围」与「管理员模式可补充的信息」。

## 风险与合规

- 结束进程可能导致未保存数据丢失；界面与文档必须反复强调。
- 杀毒软件可能拦截「枚举系统句柄」行为；可提供数字签名与误报说明渠道。
- 开机自启动必须由用户显式开启，并提供清晰关闭入口，避免被视为不透明的常驻程序。
- 自动识别设备时要避免误把系统盘、固定硬盘或同控制器下的其他设备当作可弹出目标。
- 若集成第三方工具，需遵守其许可协议。

1. **MVP**：托盘/简单窗口，自动识别可移动盘，可配置开机自启动，一键弹出，弹出失败后扫描占用进程，无需管理员时降级提示。
2. **v0.2**：管理员模式完整枚举；导出；基础的「结束进程」确认流。
